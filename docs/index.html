1. System overview

THE WEST is a browser-based hotel operations console that simulates guest intake, room assignment, current occupancy, future bookings, a 30-day room timeline, and a small REST-like API surface. It runs entirely in the browser, using IndexedDB (via Dexie.js) for persistence—no backend required.

The prototype models a hotel as a fixed set of rooms, each with one active booking record. Future bookings overwrite the room’s record, allowing the system to simulate forward occupancy pressure without introducing multi-booking complexity. The UI is terminal-inspired, prioritizing immediacy, low friction, and high visibility for operators.

Core goals:

  • Provide a fast, local, persistent guest registry.
  • Model occupancy using date-based logic instead of record existence.
  • Expose a clear operational snapshot: now, future, total.
  • Support “Generate Full” and future booking simulation.
  • Keep everything in a single, inspectable HTML file.

2. Data model

2.1 Booking record

Each room has one active booking record at any time, shaped roughly as:

{
  "room": 101,
  "name": { "first": "Ava", "last": "Morris" },
  "phone": "555-0101",
  "zip": "85001",
  "notes": "VIP arriving late",
  "flags": ["VIP"],
  "checkIn": "2026-03-12T15:00",
  "checkOut": "2026-03-14T11:00"
}

2.2 Dexie schema

The Dexie schema is intentionally simple:

db.version(3).stores({
  guests: "room"
});
  • Primary key: room
  • Exactly one record per room.
  • Future bookings overwrite the room’s record.
  • The 30-day timeline uses these records to render occupancy.

3. Operational model

3.1 One booking per room

The system intentionally avoids multi-booking per room. Each room has a single record representing either the current stay or the next future stay. This keeps reasoning about occupancy straightforward and makes the timeline easy to interpret.

3.2 Generate Full behavior

  • First click: fills all rooms with current guests.
  • Subsequent clicks: generate future guests and overwrite room records.
  • Total records stay roughly equal to the number of rooms.
  • The timeline shows future occupancy based on the latest record per room.

3.3 Date-based availability

A room is considered occupied today if:

today >= checkIn AND today < checkOut

Availability is defined as “not occupied today”—past and future bookings do not block assignment.

3.4 Status counts

The header shows a three-part snapshot:

IndexedDB • {now} now / {future} future / {total} total
  • now: guests currently in-house today.
  • future: guests whose check-in date is after today.
  • total: all booking records in IndexedDB.

4. UI components

4.1 Terminal header

Displays a pseudo-shell prompt, subsystem label, IndexedDB status, now/future/total counts, and quick actions:

  • Generate Full / Future: generate current or future hotel state.
  • API Panel: open the simulated REST console.

4.2 Guest intake form

Fields:

  • First name, last name
  • Phone, ZIP
  • Check-in, check-out (with sensible defaults if omitted)
  • Notes
  • Flags (stored as an array of strings)
  • Room selector (only rooms free today are assignable)

4.3 Registry list

Shows all bookings, sorted by room and check-in date. Each entry displays guest name, room, phone, flags, stay dates, and a notes preview. Clicking an entry opens the edit panel for that booking.

4.4 Room grid

A compact grid of rooms shows current-day occupancy at a glance. Rooms occupied today are highlighted; free rooms remain neutral. This grid is driven by the same date-based logic as the room selector and availability API.

4.5 30-day timeline

An ASCII-style timeline renders 30 days from today for each room, using blocks for occupied days and light characters for free days: 

Room 101 | ███░░░░░░░░░░░░░░░░░░░░

This gives a quick visual sense of current and future occupancy pressure.

4.6 System log

A scrolling log captures operational events: booking creation, updates, deletions, API calls, and generator actions. Entries are timestamped and prepend to keep the latest activity visible.

5. API panel (simulated REST)

The right-side API panel simulates a small REST API over the same Dexie-backed data. It’s designed for inspection, testing, and future integration thinking.

5.1 Endpoints

  • GET /guests
  • GET /rooms/available
  • POST /guests
  • PUT /guests/:id
  • DELETE /guests/:id
  • GET /search?q=
  • GET /timeline
  • GET /export
  • POST /import

5.2 Request / response

  • Request: free-form JSON or query string, depending on endpoint.
  • Response: pretty-printed JSON or raw export string in a resizable textarea.

All endpoints are thin wrappers around Dexie operations, making the API panel a live window into the same state the UI uses.

6. Availability logic

The system’s notion of availability is fully date-based. A room is unavailable only if it is occupied today; future or past bookings alone do not block assignment.

Canonical logic:

today >= checkIn AND today < checkOut

This logic is shared across the room selector, room grid, and GET /rooms/available endpoint, keeping the operational story consistent.

7. Edit panel

The edit panel allows operators to adjust an existing booking’s guest details, notes, flags, and dates. Room number remains fixed under the one-booking-per-room model. Actions include:

  • Update guest details and stay dates.
  • Adjust flags and notes.
  • Delete the booking entirely.

8. Future directions

Potential expansions on top of this prototype include:

  • Multi-booking per room with overlap detection.
  • Turnover windows and housekeeping scheduling.
  • Multi-property or multi-wing support.
  • Operator roles and permissions.
  • WebSocket-based shared timeline for multiple operators.
  • Forecasting and occupancy pressure modeling.
  • Export to CSV, ICS, or external PMS systems.

9. Philosophy

THE WEST is built around clarity, operational truth, and minimal friction. It favors a terminal-style aesthetic, local persistence, and fast iteration over heavy frameworks or opaque backends. The prototype is intentionally small, inspectable, and honest about the world it models.

It serves as a foundation for deeper operational intelligence: forecasting, turnover modeling, and eventually multi-agent collaboration around a shared, truthful state of the hotel.

10. API Samples

This section provides example requests and responses for every simulated REST endpoint in THE WEST. All endpoints operate directly on the Dexie-backed IndexedDB store.

10.1 GET /guests

Retrieve all guest booking records.

GET /guests
{
  "room": 101,
  "name": { "first": "Ava", "last": "Morris" },
  "phone": "555-0101",
  "zip": "85001",
  "notes": "VIP arriving late",
  "flags": ["VIP"],
  "checkIn": "2026-03-12T15:00",
  "checkOut": "2026-03-14T11:00"
}

10.2 GET /rooms/available

Return rooms not occupied today.

GET /rooms/available
[102, 104, 110, 122]

10.3 POST /guests

Create a new booking.

POST /guests
{
  "room": 110,
  "name": { "first": "Rae", "last": "Khan" },
  "phone": "555-0110",
  "zip": "85010",
  "notes": "Requested DND",
  "flags": ["DND"],
  "checkIn": "2026-03-12T15:00",
  "checkOut": "2026-03-14T11:00"
}
{
  "success": true,
  "room": 110
}

10.4 PUT /guests/:id

Update an existing booking.

PUT /guests/110
{
  "room": 110,
  "name": { "first": "Rae", "last": "Khan" },
  "phone": "555-0110",
  "zip": "85010",
  "notes": "Extended stay",
  "flags": ["DND"],
  "checkIn": "2026-03-12T15:00",
  "checkOut": "2026-03-16T11:00"
}
{
  "success": true,
  "updated": 110
}

10.5 DELETE /guests/:id

Delete a booking.

DELETE /guests/110
{
  "success": true
}

10.6 GET /search?q=

Search guests by name, phone, room, notes, or flags.

GET /search?q=VIP
[
  {
    "room": 101,
    "name": { "first": "Ava", "last": "Morris" },
    "flags": ["VIP"]
  }
]

10.7 GET /timeline

Return 30-day occupancy map.

GET /timeline
{
  "days": ["03-12", "03-13", "03-14", "..."],
  "rooms": {
    "101": ["█","█","░","..."],
    "102": ["░","░","░","..."]
  }
}

10.8 GET /export

Export all booking data as JSON.

GET /export
"[{ ... full JSON export ... }]"

10.9 POST /import

Import a full booking dataset.

POST /import
[

    "room": 101,
