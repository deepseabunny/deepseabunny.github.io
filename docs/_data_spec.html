THE WEST — Core Data Specification

System Overview

Provide a deterministic, inspectable, local data model representing a hotel as a fixed set of rooms with one active booking record per room. Occupancy is derived from dates; the data layer is the single source of truth for any UI or simulated REST surface.

Data Model

{
  "room": 101,
  "name": { "first": "Ava", "last": "Morris" },
  "phone": "555-0101",
  "zip": "85001",
  "notes": "VIP arriving late",
  "flags": ["VIP"],
  "checkIn": "2026-03-12T15:00",
  "checkOut": "2026-03-14T11:00"
}
Field Type Required Notes
room integer Yes Primary key; must exist in room list
name.first string Yes Non‑empty
name.last string Yes Non‑empty
checkIn ISO datetime Yes Local ISO string
checkOut ISO datetime Yes Must be after checkIn
flags array No Tags (VIP, DND)

Core Invariants

  • One Booking Per Room — overwrite semantics; no implicit history.
  • Date-Based Occupancy — day-level rule: D ≥ checkIn and D < checkOut.
  • Availability — only "occupied today" blocks assignment.
  • Determinism — timeline and counts are pure functions of stored records and clock.
  • Atomic Writes — generator/import must be transactional.

Timeline Math

Timeline window: default 30 days starting at local today. For each day D (midnight local), a booking occupies D if booking interval [checkIn, checkOut) intersects day interval [D, D+1day). Render with █ for occupied and ░ for free.
{
  "days": ["03-12","03-13","03-14","..."],
  "rooms": {
    "101": ["█","█","░","..."],
    "102": ["░","░","░","..."]
  }
}

API Semantics

  • GET /guests — return all booking records.
  • GET /rooms/available — rooms not occupied today.
  • POST /guests / PUT /guests/:id — validate then write (overwrite).
  • DELETE /guests/:id — remove record.
  • GET /timeline — deterministic timeline response.
  • POST /import — validate all records; write transactionally or abort.

Validation & Error Codes

  • Rules: room ∈ roomList; name.first/name.last non-empty; checkIn/checkOut valid ISO; checkOut > checkIn; flags array if present.
  • Errors: ERR_INVALID_PAYLOAD, ERR_MISSING_FIELD, ERR_INVALID_ROOM, ERR_INVALID_DATES, ERR_INVALID_FLAGS, ERR_IMPORT_VALIDATION, ERR_DB_WRITE.
  • Return structured JSON: { "error": "<code>", "message": "<human message>" }.

Persistence & Migration

Recommended: IndexedDB via Dexie.js with guests store keyed by room. Bulk operations must use transactions. Maintain db.version(n) migrations and validate transformed records.

Testing Guidance

  • Unit tests: validation, boundary occupancy, timeline correctness across DST/month boundaries.
  • Integration tests: import aborts on invalid records; deterministic generator reproducibility; status counts correctness.
  • Use seeded PRNG for deterministic test data.

Extension Hooks

  • Audit log store events for operator history.
  • Multi‑booking module as separate layer; keep core invariants unless migrating.
  • Housekeeping windows and turnover fields as optional extensions.
  • Multi‑property support via propertyId partitioning.

Appendix — Implementation Notes

  • Timezone: use runtime local timezone; document for integrators.
  • Clock source: new Date(); allow injection for tests.
  • Inspectability: expose validation and schema helpers for UIs.
  • Security: core spec assumes local-only; add auth/rate-limiting when exposing network APIs.
